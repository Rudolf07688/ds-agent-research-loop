# Feature Specification: Memory-Compaction Ablation Experiment (A/B/C)

**Feature Branch**: `003-memory-compaction-ablation`

**Created**: 2026-06-13

**Status**: Draft

**Input**: User description: "Look at the research docs in the `notes/research/experiments` directory and let's start to plan this spec out" — referencing `notes/research/experiments/ablation-memory-compaction-experiment-protocol.md` (the A/B/C Memory Compaction Experiment Protocol).

## Overview

This feature turns the existing autonomous-data-scientist loop into a controlled **ablation
study** that measures how the *memory strategy* shown to the LLM agent affects its experiment
decisions. The same agent, prompts, model registry, and budget are held fixed while exactly
one variable changes — what slice of prior experiment history the agent can see at each
iteration:

- **Condition A — Recent-only**: the last `k` raw experiment records.
- **Condition B — All-raw**: every prior raw experiment record.
- **Condition C — Compacted+recent**: a periodically-regenerated structured compaction
  artifact plus the last `k` raw records.

The study runs across a suite of tabular datasets and multiple random seeds, logs every
decision with full provenance, and produces comparison metrics, plots, and paired significance
tests that evaluate three hypotheses: raw history eventually hurts decision quality (A/B
threshold), structured compaction beats full raw history (B vs C), and compaction beats simple
recency (A vs C).

## User Scenarios & Testing *(mandatory)*

### User Story 1 - Run the loop under a chosen memory strategy (Priority: P1)

As a researcher, I can run the autonomous-data-scientist loop on a single dataset with a single
random seed under one of the three memory strategies (recent-only / all-raw / compacted+recent),
and get a complete, logged experiment trajectory. The memory strategy is the *only* thing that
changes the agent's visible context; everything else is held fixed.

**Why this priority**: This is the irreducible core of the study. Without the ability to run
one condition end-to-end and observe a logged trajectory, no comparison is possible. It is also
the smallest slice that delivers value on its own: a single annotated run.

**Independent Test**: Configure the loop with `memory_strategy=all-raw`, a fixed dataset, and a
fixed seed; run the configured budget of iterations; confirm that a per-iteration log is written
recording the visible-memory identifiers, the agent's proposal, the executed config, and the
resulting metrics, and that the only difference versus `recent-only` on the same seed is the
memory slice presented to the agent.

**Acceptance Scenarios**:

1. **Given** a dataset, a seed, and `memory_strategy=recent-only` with window `k`, **When** the
   loop runs, **Then** at every iteration the agent's context contains at most the last `k` raw
   experiment records and nothing else from history, and each iteration is logged with the
   memory identifiers it saw.
2. **Given** the same dataset and seed but `memory_strategy=all-raw`, **When** the loop runs,
   **Then** the agent's context contains every prior raw record, and the prompt token count per
   decision is recorded and grows across iterations.
3. **Given** identical inputs that differ only in `memory_strategy`, **When** both runs execute,
   **Then** the fixed elements (prompt template, model registry, metric, dataset, split, seed
   policy, budget) are demonstrably identical and only the memory slice differs.

### User Story 2 - Periodic structured compaction (Condition C) (Priority: P2)

As a researcher, I can run Condition C, in which the system periodically replaces accumulated raw
history with a single structured compaction artifact (confirmed findings, failed directions,
promising directions, best-known configs, unresolved questions, next-step recommendation,
confidence, rationale) generated by the LLM, and thereafter shows the agent that artifact plus
the last `k` raw records.

**Why this priority**: Compaction is the intervention the study exists to evaluate. It builds
directly on US1's parameterized loop and is required to test H2 and H3, but US1 already delivers
a runnable, comparable A-vs-B study without it.

**Independent Test**: Run Condition C on one dataset/seed with compaction cadence `m`; confirm a
compaction artifact is generated exactly at the defined trigger points, is structured per the
required fields, is saved with source-experiment lineage, and that after generation the agent
sees the artifact plus the last `k` raw records (and not the full raw history).

**Acceptance Scenarios**:

1. **Given** Condition C with cadence `m`, **When** the iteration count reaches each multiple of
   `m`, **Then** a new compaction artifact is generated from the designated source experiments
   and persisted with its source lineage.
2. **Given** a freshly generated compaction artifact, **When** the next iteration runs, **Then**
   the agent's visible memory is exactly that artifact plus the last `k` raw records.
3. **Given** the compaction step, **When** it generates an artifact, **Then** it is produced only
   from experiments at or before the trigger point (no access to future outcomes), and the
   artifact is used unchanged until the next trigger.
4. **Given** an artifact that fails structured validation, **When** it is produced, **Then** the
   run fails fast with a clear error rather than silently continuing with malformed memory.

---

### User Story 3 - Full factorial sweep across datasets and seeds (Priority: P2)

As a researcher, I can launch a single orchestrated run that executes all three conditions across
the dataset suite and all configured seeds under a fixed per-condition budget, with each
condition starting from the same baseline (same dataset summary, no prior experiments), and
collect all trajectories into one result set.

**Why this priority**: A single condition/seed run (US1/US2) proves the mechanism; the factorial
sweep is what makes the comparison statistically meaningful. It is essential to the study's
conclusion but depends on US1/US2 being correct first.

**Independent Test**: Launch the sweep over the configured datasets, conditions, and seeds;
confirm that every (dataset × condition × seed) cell runs to its budget, that all cells share
identical fixed factors, that a failure in one cell does not corrupt or halt the others, and that
results are aggregated into a single inspectable result set keyed by those three factors.

**Acceptance Scenarios**:

1. **Given** a dataset suite, the three conditions, and a set of seeds, **When** the sweep runs,
   **Then** every (dataset × condition × seed) cell completes its budget and is recorded
   separately with its full provenance.
2. **Given** the same dataset and seed across conditions, **When** comparing cells, **Then** the
   train/validation/test split and all fixed factors are byte-for-byte identical so that memory
   strategy is the sole manipulated variable.
3. **Given** a cell that errors mid-run, **When** the sweep continues, **Then** the error is
   recorded for that cell and the remaining cells still complete.

---

### User Story 4 - Outcome metrics, comparison plots, and significance tests (Priority: P3)

As a researcher, I can produce, from the aggregated results, the predefined outcome measures
(primary and secondary), per-dataset paired comparison plots, improvement and token-growth
curves, and paired significance tests (A-vs-B, B-vs-C, A-vs-C) with effect-size confidence
intervals, written to inspectable output files.

**Why this priority**: This is the reporting layer that converts logged trajectories into the
study's findings. It is the last slice because it consumes the outputs of US1–US3, but it is what
ultimately answers the research questions.

**Independent Test**: Point the analysis step at a completed result set; confirm it emits the
primary outcome (best test-set score under budget) and the secondary outcomes per condition,
generates the comparison plots and curves, and reports paired significance tests and bootstrap
confidence intervals for the three planned comparisons.

**Acceptance Scenarios**:

1. **Given** a completed result set, **When** the analysis runs, **Then** it reports, per
   condition and dataset, the best achieved test-set score under the fixed budget and the defined
   secondary outcomes (best-validation-by-iteration, area under improvement curve, number of
   improving steps, iterations to reach 90% of final best, repetition rate, search diversity,
   prompt-token-count per decision).
2. **Given** the per-dataset paired structure, **When** significance testing runs, **Then** it
   produces paired difference results (Wilcoxon signed-rank or paired-t) and bootstrap confidence
   intervals for A-vs-B, B-vs-C, and A-vs-C.
3. **Given** the analysis outputs, **When** they are written, **Then** improvement curves, token-
   growth curves, and per-dataset paired-difference plots are saved as inspectable artifacts and
   a human-readable progress note is written under `notes/`.

---

### User Story 5 - Threshold sweep over k and m (Priority: P3)

As a researcher, I can vary the recent-window size `k` and the compaction interval `m` across a
configured grid to produce threshold curves, so the study can claim *where* raw history begins to
hurt and whether compaction shifts or removes that decline.

**Why this priority**: This is the "strongest version" extension. It strengthens the contribution
from a single A/B/C point estimate to a threshold curve, but the study is already valid and
publishable-in-shape without it, so it is the lowest priority.

**Independent Test**: Configure a grid of `k` and `m` values; run the sweep; confirm results are
recorded per (k, m) combination and that threshold curves (performance vs `k`, performance vs `m`)
can be produced from them.

**Acceptance Scenarios**:

1. **Given** a grid of `k` and `m` values, **When** the sweep runs, **Then** each combination is
   executed and recorded as a distinct, comparable cell.
2. **Given** the recorded grid, **When** analysis runs, **Then** threshold curves over `k` and `m`
   are produced from the results.

### Edge Cases

- **Empty / early history**: in the first `k` iterations there are fewer than `k` prior records;
  all conditions must show whatever exists without padding or error, and Condition C must behave
  identically to recent-only until the first compaction trigger.
- **Compaction before enough data**: if a compaction trigger fires before `m` experiments exist,
  the system must either skip compaction until enough source experiments exist or compact over
  whatever exists — the chosen rule must be deterministic and logged.
- **LLM proposes an out-of-allowlist or invalid config**: handled by the existing bounded-agency
  validation — the proposal is rejected/validated before execution and the rejection is logged;
  memory strategy must not change this behavior.
- **A single cell crashes** (training error, LLM/Vertex error, malformed output): the cell is
  recorded as failed with its error; the sweep does not abort other cells.
- **Token budget / context limit exceeded in Condition B**: when accumulated raw history makes a
  prompt exceed the model's context limit, the event must be recorded as an outcome of the
  condition (it is itself evidence for H1), not silently truncated; the chosen handling must be
  explicit and logged.
- **Reproducibility**: re-running the same (dataset × condition × seed × k × m) cell must yield the
  same executed configs and metrics given the fixed seed policy, except for irreducible LLM
  non-determinism, which is why multiple seeds are run and reported.
- **Resumability**: a sweep interrupted partway must be re-runnable without recomputing already-
  completed cells, given the inspectable per-cell state.
- **Classification vs regression datasets**: the suite mixes task types; each dataset has one
  pre-registered primary metric and the analysis must aggregate comparisons in a metric-aware,
  direction-correct way (higher-is-better vs lower-is-better).

## Requirements *(mandatory)*

### Functional Requirements

**Memory strategy (core manipulation)**

- **FR-001**: The loop MUST support a selectable memory strategy with exactly three modes —
  recent-only, all-raw, and compacted+recent — selectable per run via centralized settings / a
  run parameter.
- **FR-002**: In recent-only mode, the agent's visible memory MUST be limited to the last `k` raw
  experiment records, where `k` is configurable.
- **FR-003**: In all-raw mode, the agent's visible memory MUST include every prior raw experiment
  record for the current run.
- **FR-004**: In compacted+recent mode, the agent's visible memory MUST be the most recent
  compaction artifact plus the last `k` raw experiment records, and MUST NOT include the full raw
  history.
- **FR-005**: The memory strategy MUST be the only factor that changes the agent's visible context;
  the prompt template, output schema, model, model registry, primary metric, dataset, split, seed
  policy, and budget MUST be identical across the three modes.

**Compaction**

- **FR-006**: In compacted+recent mode the system MUST generate a compaction artifact on a fixed
  cadence of every `m` experiments, where `m` is configurable. (Token-threshold triggering is an
  optional secondary mode — see FR-024.)
- **FR-007**: The compaction artifact MUST be a structured object validated against a schema with
  the fields: confirmed findings, failed directions, promising directions, best-known configs,
  unresolved questions, next-step recommendation, confidence, and rationale.
- **FR-008**: Compaction MUST be performed only over source experiments at or before the trigger
  point; the compactor MUST NOT have access to any future experiment outcomes.
- **FR-009**: Each compaction artifact MUST be persisted with its source-experiment lineage
  (which experiments it summarized) and remain unchanged until the next trigger.
- **FR-010**: Compaction MUST reuse the existing constrained-LLM contract pattern — structured
  JSON output validated by the system; a malformed or schema-invalid artifact MUST cause a
  fast, clear failure rather than silent continuation.

**Orchestration & provenance**

- **FR-011**: The system MUST provide an orchestrated sweep that runs all three conditions across
  the configured dataset suite and configured seeds under a fixed per-condition budget.
- **FR-012**: Every condition/cell MUST start from the same baseline state: the same dataset
  summary and no prior experiments.
- **FR-013**: For each iteration of every cell, the system MUST log: dataset id, condition, seed,
  iteration number, the memory artifacts/records shown (by identifier), prompt version, prompt
  token count, the proposal (action, model family, hyperparameters, reasoning, expected
  improvement mechanism), the actually executed config, validation metrics, test metrics, runtime,
  whether it improved on the incumbent, and the compaction artifact when applicable.
- **FR-014**: Per-cell state MUST be durably persisted and the sweep MUST be resumable —
  completed cells MUST NOT be recomputed on re-run. Durable state is persisted in a single
  Postgres instance (sanctioned by constitution v3.0.0), reached via a `DATABASE_URL` supplied
  through centralized settings / `.env`.
- **FR-014a**: Persisted state MUST remain inspectable: the Postgres schema MUST be documented
  and the harness MUST be able to export per-cell trajectories, compaction artifacts, and their
  source lineage to human-readable JSON/CSV, so the experiment is debuggable and rerunnable
  without database access (constitution Principle IV). The database is an addition to, not a
  replacement for, inspectable state.
- **FR-015**: A failure in one cell MUST be recorded for that cell without aborting or corrupting
  other cells.

**Datasets & task setup**

- **FR-016**: The system MUST support a suite of 5–10 tabular datasets, each a regression or
  classification task, each with one pre-registered primary metric fixed before the run (e.g.
  RMSE for regression, macro-F1 for classification).
- **FR-017**: Each dataset MUST use one fixed train/validation/test split reused identically across
  every condition and seed.
- **FR-018**: The agent action space MUST be frozen and identical across conditions: the allowed
  model families, hyperparameter ranges, and feature/target operations are defined before the run
  and enforced by validation (consistent with the existing allowlist/bounded-agency rules).

**Outcomes, analysis & reporting**

- **FR-019**: The analysis MUST compute the primary outcome — best achieved test-set score under
  the fixed experiment budget — per condition and dataset.
- **FR-020**: The analysis MUST compute the secondary outcomes: best validation score by
  iteration, area under the improvement curve, number of improving steps, iterations to reach 90%
  of final best score, repetition rate (semantically similar experiments repeated after failure),
  search diversity (distinct model families / transformation types tried), and prompt token count
  per decision.
- **FR-021**: The analysis MUST perform per-dataset paired comparisons for A-vs-B, B-vs-C, and
  A-vs-C using a paired significance test (Wilcoxon signed-rank or paired-t) and MUST report
  bootstrap confidence intervals for the effect sizes.
- **FR-022**: The system MUST produce inspectable output artifacts: per-condition improvement
  curves, token-growth curves, and per-dataset paired-difference plots, plus a human-readable
  summary; a progress note MUST be written under `notes/` per the project's progress-communication
  practice.
- **FR-023**: All outcome measures MUST be computed in a metric-aware, direction-correct way so
  that regression (lower-is-better) and classification (higher-is-better) datasets aggregate
  correctly.

**Optional / strongest-version extensions**

- **FR-024**: The system SHOULD support an optional token-threshold compaction trigger (compact
  when estimated memory tokens exceed threshold `t`) as a secondary mode alternative to the fixed
  cadence.
- **FR-025**: The system SHOULD support a configurable grid over recent-window `k` and compaction
  interval `m` to produce threshold curves, recording each (k, m) combination as a distinct,
  comparable cell.

### Key Entities *(include if feature involves data)*

- **Experiment record**: one iteration's full result — dataset id, condition, seed, iteration
  number, proposal, executed config, validation/test metrics, runtime, improvement flag, prompt
  token count, and the identifiers of the memory shown. The atomic unit of history and of logging.
- **Memory view**: the exact slice of history presented to the agent at one iteration, determined
  by the active condition (last-`k` records, all records, or artifact + last-`k`). Distinct from
  the full history; what the agent actually "sees."
- **Compaction artifact**: a structured summary of source experiments (confirmed findings, failed
  directions, promising directions, best-known configs, unresolved questions, next-step
  recommendation, confidence, rationale) with source lineage and the trigger point that produced
  it.
- **Dataset descriptor**: metadata for one task in the suite — id, task type (regression /
  classification), feature schema, the fixed train/validation/test split, and the pre-registered
  primary metric (with its optimization direction).
- **Experiment cell**: one (dataset × condition × seed [× k × m]) unit of the factorial sweep,
  owning its budget, trajectory of experiment records, completion/failure status, and resumable
  state. Persisted as rows in Postgres keyed by its factor tuple (with JSON/CSV export per
  FR-014a).
- **Outcome summary**: per-cell and per-condition aggregated measures (primary + secondary
  outcomes) plus the paired comparison/significance results that feed the report.

## Success Criteria *(mandatory)*

### Measurable Outcomes

- **SC-001**: A researcher can run any single (dataset × condition × seed) configuration end-to-end
  and obtain a complete per-iteration log in which the only cross-condition difference is the
  memory slice shown to the agent.
- **SC-002**: For the same dataset and seed, the train/validation/test split and every fixed factor
  are verifiably identical across all three conditions (the manipulation is provably isolated to
  memory strategy).
- **SC-003**: A full sweep over the configured datasets, conditions, and seeds completes with every
  cell either finishing its budget or recorded as failed, and no cell's failure prevents other
  cells from completing.
- **SC-004**: The study reports, for each of the three planned comparisons (A-vs-B, B-vs-C, A-vs-C),
  a paired effect estimate with a confidence interval and a significance result, computed across
  the dataset suite.
- **SC-005**: For Condition C, every compaction artifact is generated only from experiments at or
  before its trigger point and is reused unchanged until the next trigger — verifiable from the
  logged source lineage (no future-outcome leakage).
- **SC-006**: Prompt token count per decision is recorded for every iteration, and the report shows
  the token-growth trajectory per condition (demonstrating, in particular, unbounded growth under
  all-raw versus bounded context under recent-only and compacted+recent).
- **SC-007**: An interrupted sweep can be resumed and completes without recomputing any cell that
  had already finished.
- **SC-008**: The three hypotheses are each evaluable from the produced outputs — i.e. the report
  contains the specific metric comparison and test needed to support or reject H1 (A/B threshold
  harm), H2 (compaction beats all-raw), and H3 (compaction beats recent-only).

## Assumptions

- **Durable state is persisted in a single Postgres instance, with inspectable export.** The
  protocol calls for saving to a database; the constitution's original "no databases" non-goal
  was deliberately amended (v3.0.0) to permit one Postgres instance scoped to this experiment's
  persistence. The factorial sweep's per-cell state, logs, compaction artifacts, and lineage are
  stored in Postgres (reached via `DATABASE_URL`), and MUST be exportable to human-readable
  JSON/CSV under the existing `state/` / `outputs/` / `notes/` conventions so inspectability and
  rerun-without-DB are preserved (Principle IV; FR-014a). Postgres is an addition to, not a
  replacement for, inspectable state.
- **Reuse the existing loop, parameterized.** Per the chosen approach, the three memory modes and
  the compaction step are added to the existing `main` loop and its `history` handling; a separate
  experiment-runner orchestrates cells (dataset × condition × seed [× k × m]) and aggregates
  results, leaving the per-iteration agent contract unchanged.
- **LLM backend is the current Vertex AI / Gemini + minimal ADK setup** (from feature 002); the
  compaction call is a third constrained-JSON contract added in the same bounded-agency style.
  This feature branches from and builds on the 002 work.
- **Failure taxonomy / manual error annotation is out of scope** for this feature (the chosen
  analysis depth is "significance tests," not the full qualitative taxonomy). It may be added
  later; the per-iteration logs are rich enough to support it retroactively.
- **Dataset suite composition** (which 5–10 tabular datasets, their splits, and primary metrics)
  is a configuration concern to be finalized in planning; the existing synthetic delivery-time
  regression dataset MAY serve as one suite member and as the development/smoke dataset. Datasets
  are assumed to be small/moderate so a full sweep is feasible offline on a single machine.
- **Budget defaults** follow the protocol's examples as starting points (e.g. ~30 iterations per
  cell, multiple seeds, recent window `k≈5`, compaction cadence `m≈10`); exact values are
  configuration set in planning and overridable per run.
- **Reproducibility is seed-policy bounded.** Model training and split selection are deterministic
  under the fixed seed policy; LLM proposal non-determinism is expected and is the reason multiple
  seeds are run and reported rather than a single trajectory.

## Dependencies

- Builds on feature **002** (Vertex AI / Gemini + minimal ADK backend, containerization); inherits
  its LLM contract and settings model.
- Reuses existing modules: the experiment loop, training/scoring, history logging, prompts/schemas,
  and centralized settings — extended, not replaced.
- **Postgres** (single instance, e.g. `postgres:17-alpine`) for durable harness persistence,
  orchestrated for local runs by the root `docker-compose.yml` and reached via `DATABASE_URL`.
  Sanctioned by constitution **v3.0.0**.
- The project **constitution** governs: structured-JSON-only LLM contracts, bounded agency (no
  arbitrary code execution), inspectable state with mandatory JSON/CSV export (now also covering
  the Postgres store), anchored synthetic-data generation, and uv-managed environment all continue
  to apply to the new compaction contract and runner.
